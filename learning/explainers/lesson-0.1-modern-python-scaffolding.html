Modern Python Scaffolding

Before a single ant moves, we lay a foundation a hiring manager would nod at: a fast, reproducible, typed, src-layout package managed by uv. This is the "brush up on modern production-grade Python" part of your mission — done properly, once.

Why this lesson, first? You already know Python. What's changed under you is the tooling. In 2020 a "real" project meant juggling venv + pip + pip-tools + setup.py + flake8 + black + isort + mypy — six tools, three config files, and a lockfile you maintained by hand. The Astral toolchain collapses most of that. Getting the skeleton right now means every later lesson (sim, MAPPO, benchmarks) lands in a clean, testable home instead of a pile of scripts.

1 · The problem uv solves

Historically, Python packaging spread one job across many tools with no shared lockfile, so "works on my machine" was the default failure mode. uv is a single Rust binary that replaces pip, pip-tools, virtualenv, and pyenv — and, crucially, it writes a lockfile (uv.lock) so your environment is reproducible down to the hash. Reproducibility isn't a nicety here: your RL results must be re-runnable, and that starts at the dependency layer.

uv
project + dependency manager + Python installer. One binary.
ruff
linter + formatter (replaces flake8/black/isort). Lesson 0.2.
ty
type checker (your Astral bet). Lesson 0.2.
pyproject.toml
one declarative file: metadata, deps, tool config.

2 · Why src-layout (not flat)

Two ways to lay out a package:

❌ Flat layout

AntWorks-RL/
├─ antworks/
│ └─ env.py
└─ pyproject.toml

Your tests can import antworks straight from the working directory — even if the package isn't actually installed correctly. Packaging bugs stay hidden until someone else installs it.

✅ src layout we use this

AntWorks-RL/
├─ src/antworks/
│ └─ env.py
├─ tests/
└─ pyproject.toml

The package can only be imported once installed (via uv pip install -e .). Your tests exercise the real installed package, catching packaging mistakes immediately. This is the production-grade default.

3 · Anatomy of pyproject.toml

One file, several tables. You'll grow this all course long; today just the spine:

[project]
name = "antworks"
version = "0.1.0"
description = "Learned multi-agent ant-colony simulation"
requires-python = ">=3.12"
dependencies = [
    "numpy>=2.0",
]

[dependency-groups]        # dev-only deps, not shipped to users
dev = ["pytest>=8", "hypothesis>=6"]

[build-system]
requires = ["hatchling"]
build-backend = "hatchling.build"

[tool.hatch.build.targets.wheel]
packages = ["src/antworks"]
Note the split: runtime dependencies (what a user of antworks needs) vs the dev dependency group (pytest, hypothesis — tools only you need). Keeping these apart is exactly the discipline that keeps a shipped package lean. uv installs dev groups by default locally, and skips them in a production install.

4 · The commands you'll run

# one-time: create the project skeleton (we'll adapt what it generates)
uv init --package --python 3.12 .

# add a runtime dependency (writes pyproject + uv.lock, installs into .venv)
uv add numpy

# add a dev-only dependency
uv add --dev pytest hypothesis

# run anything inside the managed environment
uv run python -c "import antworks; print(antworks.__version__)"

uv creates and manages .venv for you — you rarely activate it by hand; you prefix commands with uv run. The uv.lock file it writes is what makes your teammate's (and CI's) environment byte-for-byte yours.
