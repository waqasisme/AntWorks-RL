Test + CI Harness

0.2 gave you a local gate (pre-commit); 0.3 gave you a validated config. 0.4 closes the loop: a real pytest layout, your first property-based test with hypothesis, and a GitHub Actions workflow that runs lint/type/test on every push — so the gate follows the code even when you forget to run it, or push from a machine without the hook installed.

Why this lesson now? The locked design commits to a hypothesis conservation-law suite for Phase 1.7 — food, population, and pheromone totals must never silently leak or duplicate across thousands of random steps. That kind of invariant testing only works if you already trust your test *harness* (layout, discovery, fixtures) before the invariants get complex. And CI matters starting now, not later: Phase 4's MAPPO will span dozens of files: buffer, GAE, PPO-clip, actor/critic nets. A red CI check catching a broken import or failing test on push is far cheaper than discovering it mid-training-run.

1 · pytest against the installed package

CLAUDE.md's src-layout convention exists precisely so tests exercise the installed antworks_rl package (via uv's editable install) rather than a package sitting on sys.path by accident of working directory — the same class of bug that "works on my machine, fails in CI" comes from.

# pyproject.toml
[tool.pytest.ini_options]
testpaths = ["tests"]
addopts = "-ra"
uv run pytest              # discovers tests/test_*.py, *_test.py
uv run pytest -v           # verbose, one line per test
uv run pytest -k config    # run only tests matching "config"

Recall from 0.3: pytest's default discovery is test_*.py / *_test.py — a test module named anything else (like __init__.py) is silently skipped, not an error. Worth internalizing since it's an easy silent-failure mode.

2 · hypothesis — testing properties, not examples

A normal assert f(2) == 4 test checks one example. hypothesis generates many varied inputs (including adversarial edge cases: zero, negative, empty, huge) from a strategy you declare, and checks a property holds for all of them — then shrinks any failure to the smallest reproducing case.

from hypothesis import given, strategies as st

@given(st.integers(min_value=0), st.integers(min_value=0))
def test_addition_commutes(a: int, b: int) -> None:
    assert a + b == b + a

This is exactly the tool for Phase 1.7's conservation laws later: "for any sequence of random valid steps, total food + eaten food + stored food is constant" is a property, not a fixed example — hypothesis is built for that shape of claim.

3 · GitHub Actions — the gate that follows the code

Pre-commit (0.2) only runs on machines where you installed the hook. A GitHub Actions workflow runs in a clean, identical environment on every push/PR — nobody can accidentally skip it.

# .github/workflows/ci.yml
name: CI
on: [push, pull_request]
jobs:
  check:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: astral-sh/setup-uv@v3
      - run: uv sync --all-extras --dev
      - run: uv run ruff check .
      - run: uv run ruff format --check .
      - run: uv run ty check .
      - run: uv run pytest

Update: the original locked design treated ty as non-blocking in CI while it's beta. The user has since overridden that: ty stays fully blocking here — a finding gets fixed properly (or suppressed visibly and deliberately with a comment explaining why), never waved through by a soft CI gate. CLAUDE.md and the locked-design record reflect this now; mypy/pyright remain the documented fallback if a beta gap ever proves genuinely unworkable.

4 · Check yourself

Immediate feedback — click an answer.

Q1. What's the actual difference between a normal pytest assertion test and a hypothesis property test?
Not necessarily — hypothesis runs your test function many times (default 100 examples), so it's typically slower per-test, not faster.
Still needs assertions — the property itself is expressed as one.
Right — the shift is from "does this one example work" to "does this property hold for the whole input space," which is exactly what conservation-law invariants in Phase 1.7 need.
Q2. This project deliberately keeps ty check blocking in CI even though `ty` is beta. What's the trade-off being accepted?
Beta checkers absolutely produce false positives — that's precisely why the original design proposed making it non-blocking. The point here is this project chose to accept that cost instead of hiding findings behind a soft gate.
Right — it's a deliberate "fix or visibly suppress, never silently ignore" stance, overriding the original beta-caveat design from 0.2/0002-locked-design-decisions.md.
Blocking vs. non-blocking doesn't change step runtime — it changes whether a failure stops the workflow's reported status.
Q3. You write tests/test_rng.py but a colleague names theirs tests/rng_check.py. What happens under pytest's default discovery?
Right — same silent-skip failure mode as 0.3's `tests/__init__.py` mistake, just with a different wrong filename. Always confirm a new test file with `pytest -v` or `pytest --collect-only` after adding it.
Discovery is per-file against the pattern, not an all-or-nothing directory check.
pytest never errors on this — it just quietly doesn't collect the mismatched file, which is the whole danger.
