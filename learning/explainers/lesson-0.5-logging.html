Logging — stdlib + structlog + rich

0.3 gave you a validated Config, already carrying a logging.level field nobody reads yet. 0.5 makes it real: one logging setup, chosen and leveled entirely from config, that prints richly-formatted colored logs on your dev machine and structured JSON lines wherever the run needs to be machine-parsed later.

Why this lesson now? Phase 9's persistent mode logs continuous colony history to SQLite plus a per-ant event log — that only works cleanly if every log line is already structured key-value data, not a hand-formatted string you'd have to regex apart later. Get the shape right now, while there's almost no code depending on it, rather than retrofitting structure onto scattered print() calls once Phase 1's sim and Phase 4's MAPPO are both emitting them.

1 · Three layers, one pipeline

stdlib logging
The actual sink: loggers, handlers, levels. Every other tool in this lesson ultimately routes through it — third-party libraries (Hydra, PyTorch later) already log via stdlib, so this is the layer that captures everyone's output, not just yours.
structlog
Wraps logging calls with structured key-value context and a chain of processors that transform a log event before it's rendered — the last processor decides the final shape: JSON, or a colored console line.
rich
RichHandler is a stdlib logging Handler that renders records as colorized, columnar console output — the human-facing renderer for local dev runs.

The pattern: structlog's processors do the structuring, then hand off to stdlib logging's formatting/handler layer for the actual rendering — ConsoleRenderer (or a rich-backed renderer) in dev, JSONRenderer in prod, both riding the same stdlib logger underneath.

2 · Wiring structlog to stdlib logging

import logging
import structlog

structlog.configure(
    processors=[
        structlog.stdlib.add_log_level,
        structlog.processors.TimeStamper(fmt="iso"),
        structlog.stdlib.ProcessorFormatter.wrap_for_formatter,
    ],
    logger_factory=structlog.stdlib.LoggerFactory(),
    wrapper_class=structlog.stdlib.BoundLogger,
)

formatter = structlog.stdlib.ProcessorFormatter(
    processor=structlog.processors.JSONRenderer(),   # or structlog.dev.ConsoleRenderer()
)
handler = logging.StreamHandler()
handler.setFormatter(formatter)
logging.basicConfig(handlers=[handler], level=logging.INFO)

log = structlog.get_logger()
log.info("ant_spawned", ant_id=7, caste="minor")   # structured, not a format string

Note the shape of a call: log.info("event_name", key=value, ...) — the event name is a short stable string, everything else is structured context. That's what makes the JSON output queryable later instead of just readable.

3 · rich for the dev console

Swap the handler for RichHandler when you want the colorized, columnar dev experience instead of raw JSON lines:

from rich.logging import RichHandler

logging.basicConfig(
    level=logging.INFO,
    format="%(message)s",
    handlers=[RichHandler(rich_tracebacks=True)],
)

Both paths are still stdlib logging.Handlers underneath — swapping dev↔prod is a config decision (which handler/renderer to install), not a code fork.

4 · Driven by config, not a constant

0.3's LoggingConfig only has level. Extend it with a renderer (or similarly-named) field so conf/logging/{default,debug}.yaml can select JSON vs. console the same way they already select the level — then call your setup function once, at the very top of main(), before anything else logs.

5 · Check yourself

Immediate feedback — click an answer.

Q1. Why route structlog through stdlib logging at all, instead of having structlog print directly?
structlog can render standalone, but that would give you a second, disconnected logging system running alongside stdlib's.
Not a performance claim — this is about which log sources actually get captured.
Right — a unified sink is the whole point; two parallel logging systems is exactly the kind of inconsistency 0.2's tooling lessons were about avoiding.
Q2. What's structurally different about `log.info("ant_spawned", ant_id=7, caste="minor")` versus `logging.info(f"ant {ant_id} spawned as {caste}")`?
They look similar in a console, but the f-string version has destroyed the structure a JSON renderer or later SQLite event log (Phase 9) would need.
Exactly — this is *why* structured logging exists: the data survives as data, not as text you'd have to parse back out.
Both still go through Python's logging machinery either way; the difference is data shape, not speed.
Q3. Per this project's conventions, where should the choice between JSON and console (rich) rendering live?
Right — config-driven selection is the established pattern from 0.3's config groups; it also means CI/prod can force JSON while your terminal still gets rich output, from the same code path.
A TTY check is a reasonable *default*, but hardcoding it removes the ability to force JSON locally for debugging a prod-log issue, or force rich output in a CI log viewer that wants it.
Rich console output exists precisely because raw JSON is unpleasant to read live during dev — that's the whole reason this lesson wires up both.

  • LoggingConfig (in src/antworks_rl/__init__.py or wherever you've since moved config models) gains a second field alongside level — e.g. renderer: Literal["console", "json"] — with a sensible default.

  • main() calls setup_logging(...) immediately after validating Config, before any other logging happens, and emits at least one structured log call (e.g. log.info("config_loaded", seed=config.seed)) in place of the old print(config).

  • Prove level filtering works: a log.debug(...) call is silent at the default level and visible when you override to the debug config group.
