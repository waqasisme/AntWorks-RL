Config Spine — Hydra + Pydantic

0.1 gave you a package; 0.2 made it self-enforcing. 0.3 gives it a spine: every run — sim, training, sweep — is driven by one composable, typed config object instead of scattered constants or argparse flags.

Why this lesson now? The locked design says every stochastic component draws from an injected, seeded RNG "whose seed comes from config" — that requires a config object to exist before Phase 1's grid and RNG do. Later, Phase 5 needs to sweep MAPPO-vs-IPPO and other hyperparameters without editing code, and Phase 4's MAPPO has dozens of knobs (clip ratio, GAE lambda, learning rates, network sizes). Hydra composition + multirun sweeps is exactly the tool for that; Pydantic is what stops a mistyped YAML key from silently becoming None three modules deep at 2am during a training run.

1 · Two tools, two different jobs

Hydra
Composes YAML fragments (config groups) into one tree, and lets you override any leaf from the CLI — seed=7 sim.grid_size=64 — without touching a file. Also drives --multirun sweeps later.
Pydantic
Takes the composed, still-dynamic tree (an OmegaConf/DictConfig) and validates it into a typed, immutable-by-convention Python object — wrong type, missing field, or unknown key becomes a loud ValidationError at startup, not a mystery three files later.

Hydra alone gives you a dict-like DictConfig — attribute access works, but nothing stops cfg.seed = "banana" from flowing straight into your RNG constructor. Pydantic is the seam where "YAML text" becomes "a value the type checker and your code can actually trust."

2 · Config groups & composition

Hydra's unit of composition is a config group: a subdirectory of conf/ holding interchangeable YAML variants, wired together by a defaults list in the top-level config.

conf/
  config.yaml          # top-level: defaults list + any ungrouped keys
  logging/
    default.yaml        # e.g. level: INFO
    debug.yaml          # e.g. level: DEBUG

# conf/config.yaml
defaults:
  - logging: default
  - _self_

seed: 0
run_name: dev

# entrypoint
import hydra
from omegaconf import DictConfig, OmegaConf

@hydra.main(version_base=None, config_path="../conf", config_name="config")
def main(cfg: DictConfig) -> None:
    print(OmegaConf.to_yaml(cfg))

Swap groups or override leaves from the CLI without editing YAML:

uv run antworks-rl logging=debug seed=42

3 · Bridging to Pydantic

The composed DictConfig is still dynamic. Convert it to a plain container and hand it to a Pydantic model — that's the validation boundary:

from omegaconf import OmegaConf
from pydantic import BaseModel

class LoggingConfig(BaseModel):
    level: str = "INFO"

class Config(BaseModel):
    seed: int
    run_name: str
    logging: LoggingConfig

@hydra.main(version_base=None, config_path="../conf", config_name="config")
def main(cfg: DictConfig) -> None:
    config = Config.model_validate(OmegaConf.to_container(cfg, resolve=True))
    ...

Give every Pydantic model model_config = ConfigDict(extra="forbid") so a typo'd YAML key (see instead of seed) raises instead of silently vanishing.

4 · Testing config with pytest

Config is the first thing your entrypoint does — if it's wrong, nothing else runs. Test it early and loudly.

Use Hydra's compose API to load configs programmatically in tests (no CLI args needed):

from hydra import initialize, compose
from hydra.core.global_hydra import GlobalHydra
import pytest

def test_config_loads_and_validates():
    GlobalHydra.instance().clear()  # reset Hydra state
    with initialize(version_base=None, config_path="conf"):
        cfg = compose(config_name="config")
        config = Config(**OmegaConf.to_container(cfg, resolve=True))
    assert config.seed > 0
    assert isinstance(config.run_name, str)

Also test that validation fails loudly on bad input:

def test_config_rejects_bad_type():
    GlobalHydra.instance().clear()
    with initialize(version_base=None, config_path="conf"):
        cfg = compose(config_name="config", overrides=["logging.level=123"])
        with pytest.raises(ValidationError):
            Config(**OmegaConf.to_container(cfg, resolve=True))

This catches schema drift early: if someone adds a field to YAML and forgets to add it to Config, or vice versa, the test breaks before it ships.

6 · Your exercise you code this

Give the repo a real config spine. End-state:

  • conf/config.yaml exists with a defaults list and at least one config group (e.g. conf/logging/{default,debug}.yaml) — mirroring the tree shown above.
  • A Config Pydantic model (and any nested sub-models, e.g. LoggingConfig) living in the package (e.g. src/antworks_rl/config.py) that mirrors the YAML schema, with extra="forbid" set.
  • The package entrypoint uses @hydra.main to compose the config, then converts + validates it into your Config model before doing anything else.
  • Demonstrate a CLI override actually changing behavior — e.g. uv run antworks-rl seed=123 prints/logs a different seed than the default run.
  • Demonstrate the validation boundary failing loudly — pass an override that breaks a type or field name (e.g. logging.level=123 or a misspelled key) and show the ValidationError it raises, rather than a silent bad value.
  • At least one pytest test that loads the default config via Hydra's compose API (or by invoking the entrypoint) and asserts it validates into your Config model successfully.
