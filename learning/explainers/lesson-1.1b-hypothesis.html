Property-based testing with Hypothesis

Lesson 1.1 ends with two tests, and one of them is not like the others. The determinism test is ordinary pytest. The invariant test asks for something new: "over randomized valid configs, ant ids are unique, all positions are in-bounds, no cell exceeds the cap." That word — over randomized configs — is Hypothesis. This companion teaches the mental shift from "test one example" to "state a property and let the machine attack it," then walks you right up to the edge of writing test_world.py yourself.

Why this now? Your make_world has invariants that are true by construction — the slot draw can't exceed the cap, `check_ants_fit` can't let an impossible config through. When you believe code is correct by construction, an example test is nearly worthless: you'll pick the example that confirms your belief. Property-based testing does the opposite — it hunts for the input that breaks your belief, and when it finds one it hands you the smallest one. Two lessons ago, the 3-D-unravel bug was exactly this shape: invisible to a hand-picked example, obvious to a strategy that tries a 4×4 grid with a cap. This is the tool that would have caught it on example one.

1 · Two ways to test the same function

An example-based test asserts a fact about one input you chose:

def test_make_world_places_all_ants():
    cfg = WorldConfig(width=6, height=6, n_ants=4, n_food=3, occupancy_cap=2)
    world = make_world(cfg, np.random.default_rng(0))
    assert len(world.ant_ids) == 4        # true for THIS config

It passes. But you proved one point in a vast input space, and you chose the point. A property-based test asserts a fact that must hold for every input in a space you describe — and lets Hypothesis pick the points, adversarially:

from hypothesis import given, strategies as st

@given(n_ants=st.integers(min_value=1, max_value=500))
def test_all_ants_get_ids(n_ants):
    cfg = WorldConfig(width=100, height=100, n_ants=n_ants, n_food=0, occupancy_cap=3)
    world = make_world(cfg, np.random.default_rng(0))
    assert len(world.ant_ids) == n_ants   # true for ALL n_ants in range

The shift is from "does it work for 4?" to "what is always true, and for which inputs?" You stop writing answers and start writing invariants — statements of the form "no matter the valid config, X holds." That reframing is the whole skill; the library is just the engine that runs it.

The invariant-spotting habit. Good properties are usually one of a few shapes. Learn to reach for them:
  • Never-exceeds / bounds — "no cell holds more than occupancy_cap", "every position is inside the grid." Yours are mostly these.
  • Conservation / counts — "exactly n_food cells hold food", "len(ant_ids) == n_ants." The thing you put in equals the thing you get out.
  • Uniqueness — "no two ants share an id."
  • Round-trip — "serialize then deserialize returns an equal object." (You'll want this when StateSnapshot lands in 1.6.)

2 · The three pieces

Every Hypothesis test is the same three parts. Name them and the syntax stops being magic:

1 · @given(...)
The decorator that turns one test function into hundreds of runs. Each keyword maps a parameter to a strategy. Hypothesis calls your function many times with drawn values (100 by default).
2 · a strategy
A recipe for generating (and shrinking) values. st.integers(), st.floats(), st.lists(...), st.booleans(). Strategies compose — that's the power.
3 · the assertion
The property, as a plain assert. If it ever fails for any drawn value, Hypothesis reports the failure — after shrinking it to the minimal case.

The rng is worth a beat. Your make_world takes an injected np.random.Generator — so the config is what Hypothesis should vary, and the seed can stay fixed inside the test. You're testing "for all valid configs, the invariants hold," not "for all RNG streams." (You can vary the seed too with st.integers() for the seed — a nice belt-and-braces, but not required.)

3 · Strategies, and the one that matters here: @composite

Simple strategies are drawn independently. That breaks the moment your inputs have a relationship — and yours do. Watch what goes wrong if you draw the five config fields independently:

@given(
    width=st.integers(1, 20), height=st.integers(1, 20),
    n_ants=st.integers(1, 10_000), occupancy_cap=st.integers(1, 8),
)
def test_bad(width, height, n_ants, occupancy_cap):
    cfg = WorldConfig(...)   # ValidationError: n_ants exceeds max occupancy

Most draws are unsatisfiable — a 2×2 grid with cap 1 holds 4 ants, but Hypothesis just drew n_ants=9000. Your own check_ants_fit rejects it, and the test dies on setup instead of testing anything. You need n_ants drawn after, and from, the grid dimensions. That's what @composite is for — a strategy that draws values in sequence, each depending on the last:

from hypothesis import strategies as st

@st.composite
def valid_world_configs(draw):
    width  = draw(st.integers(min_value=1, max_value=20))
    height = draw(st.integers(min_value=1, max_value=20))
    cap    = draw(st.integers(min_value=1, max_value=8))
    # now n_ants and n_food are drawn from a range the grid can actually satisfy
    n_ants = draw(st.integers(min_value=1, max_value=width * height * cap))
    n_food = draw(st.integers(min_value=0, max_value=width * height))
    return WorldConfig(width=width, height=height,
                       n_ants=n_ants, n_food=n_food, occupancy_cap=cap)

@given(cfg=valid_world_configs())
def test_world_invariants(cfg):
    world = make_world(cfg, np.random.default_rng(0))
    ...   # your assertions

Read the shape, not the syntax: draw(...) pulls one value from an inner strategy, and because n_ants's upper bound is width*height*cap, every config this produces is satisfiable by construction. No wasted examples, no setup crashes — the same "make the bad state unrepresentable" instinct you applied to occupancy, now applied to test inputs. This is the single most important Hypothesis technique for your project; almost every sim invariant test you write from here will start with a @composite config strategy.

Prefer max_value over .filter(). A tempting shortcut is to draw n_ants broadly and .filter(lambda n: n <= w*h*cap). Resist it. Filtering discards draws that fail the predicate, and near saturation it discards almost all of them — Hypothesis raises a FailedHealthCheck for filtering too much. Deriving the bound (as above) generates only valid values. Construct, don't reject — the same lesson as your ant placement.

4 · The payoff: shrinking

Here's what makes Hypothesis worth learning over "just loop with random." When a property fails, it doesn't hand you the wild 17×13-cap-6 config that happened to trip it. It shrinks: re-runs the property against progressively simpler inputs, keeping only ones that still fail, until it finds the minimal counterexample — then prints that.

Falsifying example: test_world_invariants(
    cfg=WorldConfig(width=1, height=1, n_ants=2, occupancy_cap=1, ...),
)

That's a debugging gift. Instead of "it broke somewhere in a 200-ant world," you get "it breaks with 2 ants on a 1×1 grid" — the smallest possible reproduction, which usually points straight at the cause. And Hypothesis remembers failures in a local .hypothesis/ database, so the next run replays the known-bad example first. Fix the bug, the replay confirms it; a green run means the counterexample is genuinely gone.

Mental model: example tests prove presence of behaviour for a point you trust; property tests attack a claim across a space you distrust, and shrink any breach to its core. You want both. Determinism is a fact about specific seeds → an example test. Invariants are claims about all configs → property tests.

5 · The three gotchas that bite NumPy users

GotchaWhat happens / what to do
deadlineHypothesis fails a test that runs "too slow" (200 ms default per example) to catch perf cliffs. A large grid draw can trip this spuriously. Cap your dimensions small (≤20) so worlds are cheap, or set @settings(deadline=None) on the test.
Health checksfilter_too_much / data_too_large fire when your strategy is wasteful. The fix is almost always to derive bounds (§3), not to suppress the check.
Determinism vs replayHypothesis varies its own examples run-to-run by design (it's exploring). That's separate from your seeded rng. Keep the sim seed fixed inside the test; let Hypothesis vary the config. If you need fully reproducible example selection in CI, @settings(derandomize=True).

@settings stacks above @given:

from hypothesis import given, settings

@settings(max_examples=200, deadline=None)
@given(cfg=valid_world_configs())
def test_world_invariants(cfg):
    ...

6 · Mapping it onto your 1.1 deliverables

The exercise names two tests. Here's which tool each wants and why:

DeliverableToolWhy
Same seed ⇒ identical worlds; different seeds differplain pytestA claim about specific seeds, not a space. Build twice, compare. Use your __eq__ — and also assert np.array_equal(a.ant_ids, b.ant_ids), because __eq__ doesn't check ids.
Ids unique · positions in-bounds · no cell over cap · counts matchHypothesisClaims about all valid configs. One @composite strategy + one test with four asserts.
Impossible config raisesplain pytestpytest.raises(ValidationError) on an n_ants > w·h·cap config. One specific bad input.

For the "no cell over cap" assert, the vectorized check is worth knowing: flatten each position to a cell index and np.bincount it — the max count is the fullest cell.

# positions (n,2) -> per-cell occupancy, without a Python loop
flat = world.ant_positions[:, 0] * world.height + world.ant_positions[:, 1]
counts = np.bincount(flat, minlength=world.width * world.height)
assert counts.max() <= world.occupancy_cap

No such limit — st.integers() is unbounded by default. The issue is the relationship between fields, not a ceiling.
Q2. A property test fails. What does Hypothesis give you that a hand-rolled random loop doesn't?
No — a passing property test is strong evidence, never a proof. It samples the space; it doesn't exhaust it.
The opposite — Hypothesis deliberately varies examples to explore. It's your sim seed that stays fixed, not Hypothesis's search.
Exactly — shrinking hands you the smallest reproduction, and the database replays it until you've fixed it.
Q3. Which of your 1.1 tests should not be a Hypothesis test?
Right — determinism is an example-shaped claim (build twice, compare). Save Hypothesis for the "for all valid configs" invariants.
Cap-respect must hold across all configs — that's the textbook property test. Determinism is the example one.
Uniqueness is a classic property (len(set(ids)) == len(ids)) — very much expressible.

8 · Your exercise — tests/test_world.py the real deliverable

This is the 1.1 test file, now with the tool to write it. Home: tests/test_world.py (flat under tests/, unique basename — the layout convention from 0.4). Build it up in this order:

  • Determinism (pytest). Two worlds, same seed → assert a == b and np.array_equal(a.ant_ids, b.ant_ids). A third world, different seed → assert a != b. (Use a small grid so different seeds reliably diverge.)
  • The strategy. Write a @st.composite valid_world_configs(draw) that draws width, height, cap first, then n_ants ∈ [1, w·h·cap] and n_food ∈ [0, w·h]. Return a WorldConfig.
  • Invariants (Hypothesis). One @given(cfg=valid_world_configs()) test asserting: ids unique (len(set(...)) == n_ants); every position in-bounds; per-cell occupancy ≤ cap (the bincount trick); food.sum() == n_food; len(ant_ids) == n_ants.
  • Edges as their own cases. Full saturation n_ants == w·h·cap (the draw must still succeed); n_ants == 1; n_food == 0.
  • The rejection case. with pytest.raises(ValidationError): WorldConfig(width=2, height=2, n_ants=99, occupancy_cap=1, ...). Import it from pydantic.
  • Green gates. uv run ruff check ., uv run ty check ., uv run pytest — all pass. Watch the run count jump: one @given test is hundreds of cases.
Then say "check now." I'll review the strategy first — is every generated config satisfiable, or are you leaning on .filter()? — then the assertions, then whether the edges are real cases or hand-waved. If an invariant is genuinely true-by-construction, I may suggest temporarily breaking make_world to watch the test go red — a property test you've never seen fail is a property test you don't yet trust.
