AntWorks-RL · Phase 1 · Lesson 1.1

Grid & entities

Phase 0 gave you a typed, tested, config-driven, logging-wired skeleton. Now the simulation begins. This lesson builds the world's data model — the grid, the ants, the food, the nest — and the two decisions that will quietly govern everything downstream: how ant state is laid out in memory, and how an ant keeps its identity when its array row does not. No movement yet, no pheromones, no energy — just a correct, seeded, headless world at t = 0.

Why this lesson now? Everything in Phase 1 — movement, occupancy, energy, the 5-channel pheromone field, the StateSnapshot — reads and writes this data structure. Get the layout right and the vectorized updates in 1.2–1.4 are a few lines of NumPy; get it wrong (a list of Ant objects) and you rewrite the core the moment you scale past a few dozen ants. The MISSION is emergent foraging from thousands of learned agents — the data model is where that scale is won or lost, and it is far cheaper to decide now than to retrofit later.

1 · The world is sim state, not a picture

Our locked design (learning-records/0002) is a grid world: a 2-D lattice, ants take 8-directional moves + stay, cells have occupancy caps. The realism budget goes into pheromone chemistry and behaviour later — not locomotion physics. So the world at this stage is deliberately small: a grid size, a nest, some food, some ants with positions.

Hold two conventions from day one, because they are painful to add back later:

Headless & serializable
The world is plain data (NumPy arrays + scalars). It knows nothing about pygame, colours, or the screen. A renderer will later consume a StateSnapshot derived from it (Lesson 1.6) — never the reverse.
Seeded & injected RNG
Every random choice — nest placement, food scatter, ant spawn — draws from an injected np.random.Generator (your make_rng from 0.x), seeded from config. No global np.random. Same seed ⇒ byte-identical world.

2 · The layout decision: Struct-of-Arrays

The instinct from OO Python is a class per ant:

class Ant:            # ← the tempting, wrong-for-us choice
    def __init__(self, id, x, y):
        self.id = id; self.x = x; self.y = y
ants = [Ant(...) for _ in range(n)]   # Array-of-Structs (AoS)

This is Array-of-Structs: each ant's fields sit together, and the colony is a Python list of objects. It reads nicely and vectorizes terribly. To move every ant you iterate in Python and chase pointers all over the heap.

The alternative is Struct-of-Arrays (SoA): one contiguous array per field, indexed by row.

pos   = np.zeros((n, 2), dtype=np.int32)   # column: x, y  for every ant
ids   = np.zeros(n,       dtype=np.int64)   # stable identity per row
alive = np.ones(n,        dtype=bool)       # is this row a live ant?

Now "move all live ants north" is pos[alive, 1] -= 1 — one vectorized NumPy op over contiguous memory, no Python loop. This is the data-oriented choice: lay out memory for the access pattern (update one field across all entities), not for human mental models. On a particle sim of exactly this shape, SoA measured up to ~40× faster than AoS (Dembinski). For a colony you want to scale to thousands, that is the difference between real-time and a slideshow.

Don't reach for numpy structured (record) arrays either — np.zeros(n, dtype=[('x',int),('y',int)]) looks like the best of both, but it lays out memory as AoS again (fields of one ant interleaved), so it loses the cache/SIMD win. Plain parallel arrays (or a small dataclass holding those arrays) is the deep-module move: a simple interface over a layout chosen for speed.

3 · Identity ≠ row index

Here is the subtle one. In 1.3 ants will die, and in later phases they'll be born mid-episode. If ant identity is its row index, then compacting the arrays (or reusing a dead row for a newborn) silently gives a new ant an old ant's identity — and your per-ant event log, your narrative hooks, and your renderer's "follow that ant" all corrupt.

So identity gets its own field, independent of storage position:

  • A monotonic counter mints a fresh id for every ant ever created; it never rewinds. Row 4 today and row 4 after a death-compaction are different ants with different ids.
  • Lookups go through an id → row indirection (a dict, or a search) so external references are by id, internal vectorized work is by row.

This is exactly the game-engine entity-handle pattern (bitsquid / ECS): the row is where the data lives right now; the id is who this is, forever. You don't need generational bit-packing yet — a plain growing counter is enough for 1.1 — but keep the two concepts separate from the first line of code.

4 · Occupancy: a rule, enforced at construction

Cells have an occupancy cap (design decision 0002). Movement enforcement is Lesson 1.2's job — but even at t = 0 your placement must respect it: don't spawn 10 ants and the nest on the same cell if the cap is 4. Placement becomes "draw candidate cells from the injected RNG, reject ones already full." Getting this invariant true at construction is what your tests will pin down.

Keep scope tight: no movement, no energy, no pheromone field, no stepping the world. Those are 1.2–1.4. Lesson 1.1 is the static, seeded world at time zero and the invariants that must hold there.

6 · Your exercise you code this

Build the static, seeded world at t = 0. Suggested home: a new src/antworks_rl/world.py (a plain dataclass holding NumPy arrays is fine — you decide the exact shape; the acceptance criteria are about behaviour, not naming). End-state:

  • A config group for the world (extend the existing Hydra/Pydantic spine — e.g. a world group validated by a Pydantic model): grid width/height, n_ants, n_food, and the per-cell occupancy_cap. Overridable on the CLI like logging=debug already is.
  • A world data structure laid out Struct-of-Arrays: at minimum an ant positions array (n, 2), an ids array, and an alive mask — plus the nest position and food positions. No per-ant Python objects in the hot state.
  • A constructor / factory make_world(config, rng) that takes an injected np.random.Generator and places nest, food, and ants using only that generator — no global random/np.random.
  • Stable identity: each ant gets a unique id from a monotonic counter (id is not the row index), and there's a way to resolve id → row.
  • All placement respects bounds and the occupancy cap at construction: every entity is inside the grid, and no cell holds more than occupancy_cap ants.
  • Determinism test (pytest): two worlds built from the same seed are element-wise identical (positions + ids); two different seeds differ.
  • Invariant test (hypothesis, over randomized valid configs): ant ids are unique; all positions are in-bounds; no cell exceeds the cap; counts match the config. Let Hypothesis pick width/height/n_ants/n_food/cap within ranges that are actually satisfiable.
  • The world stays headless: no import of any renderer, no drawing. It's plain data an eventual StateSnapshot can read.
  • Green gates: uv run ruff check ., uv run ty check ., and uv run pytest all pass.
Then come back to the chat. Tell me you're done, or paste world.py + the tests + the config change. I'll review the layout choice, the id/row separation, and the seeding — and specifically stress the invariants (what happens when n_ants can't fit under the cap? does your placement loop terminate?) — before we open Lesson 1.2 (Movement & occupancy), where these arrays finally start to move.
