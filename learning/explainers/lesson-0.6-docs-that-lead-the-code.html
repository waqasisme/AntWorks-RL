AntWorks-RL · Phase 0 · Lesson 0.6

Docs that lead the code

Phase 0 is nearly done: you have a typed, tested, config-driven, logging-wired skeleton. This lesson closes it by making the docs honest — formalizing the CLAUDE.md that was seeded on day one with the conventions you actually established in 0.1–0.5, and writing a lean, clearly-labelled README.md that says what the project is, what it does today, and what is still just vision.

Why this lesson now? This is a portfolio project (see MISSION): the README is the first thing a hiring manager reads, and nothing damages credibility faster than a README describing a running ant colony that doesn't exist yet. Phase 0 is exactly the right moment — there are now real conventions worth writing down, but little enough code that documenting it takes minutes, not a retrospective archaeology dig. Get the docs-track-reality habit locked before Phase 1's simulation multiplies the surface area.

1 · The one rule: docs track reality

This repo already has a stated principle (you applied it yourself back in 0.1 — see learning-records/0003): documentation describes what the code actually does, never what you wish it did. Intent and plans are legitimate — but they live in clearly-labelled vision/roadmap sections, never disguised as as-built fact.

The failure mode this prevents. "Aspirational docs" read as lies the moment a reader runs the code and it doesn't match. A README that says "watch ants forage with uv run antworks-rl" when that command just logs a config and exits is worse than no README — it burns trust. The fix is a discipline, not a tool: every concrete claim must be true today, and everything else is explicitly flagged as not-yet.

This isn't a personal quirk — it's the same instinct behind Architecture Decision Records (which your learning-records/ already are): capture why, append-only, tied to what really happened. Docs earn their keep by being trustworthy.

2 · Two documents, two jobs

The Diátaxis framework is a useful lens: documentation serves different needs, and mixing them makes all of them worse. For Phase 0 you're writing two docs with sharply different audiences:

README.md
Audience: a human landing on the repo cold. Orientation. What is this, what works today, how do I run what exists, where do I go next. Short. Honest status banner up top.
CLAUDE.md
Audience: Claude (and future you) working in the repo. The conventions the code follows, so contributions match. Reference material — the "how we do things here" that 0.1–0.5 established.

ARCHITECTURE.md is deliberately not written yet — there's no real structure to describe until Phase 1 exists. Documenting an architecture you haven't built would violate rule #1. It's scheduled for the end of Phase 1, on purpose.

3 · The conventions to capture

Your CLAUDE.md was seeded early with a "grow this list through Phase 0" note. Now you cash that in: fold in the conventions you actually established, each already backed by a decision in learning-records/. The real ones from 0.1–0.5:

  • src-layout, tests exercise the installed package (0.1) — and the package is antworks_rl, not antworks (0003).
  • Astral toolchain + gates: uv / ruff / ty; ty is blocking in CI — findings get fixed or visibly suppressed, never silenced with a non-blocking gate (0005).
  • Config spine: Hydra composition → Pydantic validation; config groups (e.g. logging=debug); Hydra's relative config_path resolves as a module, needing conf/__init__.py (0004).
  • Injected, seeded RNG everywhere — no global random/np.random.
  • Structured, config-driven logging (0.5): event + key-value fields, JSON vs console by config; and the caveat that Hydra owns the root logger, so setup must force its handlers (0006).
  • Flat test layout: unique-basename modules directly under tests/, no stdlib-shadowing test packages (0007).

Don't invent conventions here — only write down what the code and the learning-records already show to be true. If you're tempted to document something the code doesn't do yet, that's a roadmap item, not a convention.

4 · The status banner

The single most important honesty device in the README is a status line near the top that sets expectations before anyone runs anything:

> **Status: Phase 0 (foundation) complete — not yet a runnable simulation.**
> The toolchain, config, logging, and test harness are in place; the ant
> simulation itself begins in Phase 1. See `learning/ROADMAP.md` for the plan.

Now the "vision" section below it can describe the full learned-stigmergy ant colony in all its glory — because the reader already knows that's the destination, not the current stop. Label it ("Vision" / "Where this is going") so the boundary between real and planned is unmissable.
