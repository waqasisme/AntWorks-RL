Python dataclasses

A side-quest, but a well-timed one. Lesson 1.1 tells you "a plain dataclass holding NumPy arrays is fine" for the world — so before you write World, this lesson makes that choice deliberate. You already reach for pydantic.BaseModel for config (0.3). The skill here is knowing which of the two to reach for, when, plus the four decorator switches — slots, frozen, field, __post_init__ — that separate a toy dataclass from a production one. And one NumPy trap that will bite your 1.1 determinism test if you don't see it coming.

Why this now? Your world state is the hottest data in the project — every step of 1.2–1.4 reads and writes it, and the MISSION is thousands of agents. A dataclass is the right container for it precisely because it is a zero-overhead record: no validation, no coercion, just __init__/__repr__/__eq__ generated for you. But "zero-overhead" cuts both ways — the defaults (mutable-default guard, auto __eq__, no slots) have sharp edges that matter more at scale, not less. Get the switches right once here and every entity/snapshot dataclass downstream inherits the decision.

1 · What @dataclass actually generates

A dataclass is not a new kind of object — it's a code generator. You declare fields as annotated class attributes; the decorator writes the boilerplate dunder methods from those annotations at class-creation time. Nothing runs at runtime that a hand-written class wouldn't.

from dataclasses import dataclass

@dataclass
class Nest:
    x: int
    y: int
    capacity: int = 100      # field with a default

# generated for you: __init__, __repr__, __eq__
Nest(3, 4) == Nest(3, 4)     # True  — value equality, not identity

Three things to internalise, because they're the whole value proposition:

  • The annotation is what makes a field. x: int is a field; x = 5 (no annotation) is not — it stays a plain class attribute the decorator ignores. The type is not enforced at runtime; ty checks it statically, that's all.
  • Order of declaration = order of __init__ positional args. A field with a default cannot precede one without (same rule as function args).
  • __eq__ compares (self.field1, self.field2, …) as a tuple. Remember this — §5 shows how it detonates on NumPy arrays.

2 · The decision: dataclass or Pydantic?

You now have both in the codebase. They look similar and solve opposite problems. The rule of thumb:

Pydantic BaseModel → boundaries
Data crossing into your program from an untrusted place — YAML config, a request, a file. It validates and coerces ("4"→4), raises on bad input, and costs real cycles per construction. That's exactly what you want for Config: fail loud at load time. It's a gate.
@dataclass → internal state
Data you already trust, built and mutated in your own hot loop — the World, an entity record, a StateSnapshot. No validation, no coercion, no per-instance overhead beyond a normal object. It's a struct, not a gate.

Concretely for AntWorks: Config stays BaseModel (it parses config you might fat-finger). World should be a @dataclass — validating a million-element array on every construction, then again after every step, would tax the exact hot path 1.1 built the SoA layout to protect. Validate once at the config boundary; trust the arrays thereafter.

The trap in between: Pydantic also ships a @pydantic.dataclasses.dataclass, and stdlib dataclasses can be nested inside a BaseModel. Don't blur them by reflex. If you find yourself wanting Pydantic's validation on your world state, that's a signal the untrusted data should have been validated earlier, at the config edge — not that the hot struct needs a gate bolted on.

3 · The four switches that make it production-grade

field(default_factory=…) — the mutable-default rule

A dataclass field can't default to a mutable object. This is illegal and raises at class-definition time:

@dataclass
class World:
    food: list = []          # ValueError: mutable default not allowed

Because a bare default is shared across all instances — the classic Python footgun — dataclasses refuse it and make you say "call this factory per instance" instead:

from dataclasses import dataclass, field
import numpy as np

@dataclass
class World:
    positions: np.ndarray = field(default_factory=lambda: np.zeros((0, 2), np.int32))
    food: list[tuple[int,int]] = field(default_factory=list)

field() is also where you tune per-field behaviour: compare=False (exclude from __eq__), repr=False (hide from printout — good for a big array), kw_only=True, and metadata=.

slots=True — smaller, faster instances (3.10+)

By default every instance carries a __dict__. @dataclass(slots=True) generates __slots__ instead: less memory per instance and faster attribute access, at the cost of "no ad-hoc new attributes." For a single World holding arrays this barely matters — but it's the reason 1.1 told you not to make a dataclass per ant. A slotted per-ant object is still an object per ant; the SoA arrays win regardless. Reach for slots=True on any small record you might hold many of.

frozen=True — immutability for snapshots

A frozen dataclass blocks attribute assignment after construction (__setattr__ raises) and, as a bonus, becomes hashable. This is the right default for StateSnapshot in 1.6: a renderer should never mutate the frame it was handed.

frozen is shallow. It freezes the binding, not the object. A frozen dataclass holding a NumPy array still lets you do snapshot.positions[0] = 99 — the array is mutable, only the name positions is locked. If a snapshot must be truly read-only, set arr.flags.writeable = False on the arrays too. Frozen ≠ deep-frozen.

__post_init__ — derived fields & the frozen escape hatch

__init__ is generated, so your hook for "compute something after the fields are set" is __post_init__. Use it for a derived index or a cheap invariant assert:

@dataclass
class World:
    ids: np.ndarray
    positions: np.ndarray
    _id_to_row: dict[int,int] = field(init=False, repr=False)  # not a constructor arg

    def __post_init__(self):
        self._id_to_row = {int(i): r for r, i in enumerate(self.ids)}

On a frozen instance even __post_init__ can't assign normally — you must go around your own lock with object.__setattr__(self, "field", value). Needing that a lot is a hint the field should be a real constructor arg instead.

4 · A couple more you'll want

SwitchWhat it does / when
kw_only=TrueAll fields become keyword-only in __init__ (3.10+). Frees you from the "defaults must come last" ordering rule and makes call sites self-documenting — nice for a World with many params.
order=TrueGenerates __lt__ etc. from the field tuple. You rarely want it on world state; handy for small sortable records.
dataclasses.replace(obj, x=…)Returns a copy with some fields changed — the idiomatic way to "mutate" a frozen snapshot: build the next one.
asdict() / astuple()Recursively convert to plain dict/tuple. Convenient — but it deep-copies, so don't call it on your million-element arrays every frame.

5 · The trap that will hit your 1.1 test

Your 1.1 determinism test asserts "two worlds from the same seed are identical." If World is a plain dataclass with NumPy-array fields, the obvious assertion raises — it doesn't just return False:

world_a == world_b
# ValueError: The truth value of an array with more than one element is ambiguous.

Why: the generated __eq__ compares field tuples. Tuple comparison evaluates positions_a == positions_b, but for NumPy that returns an array of bools, and Python then tries bool(that array) — which NumPy refuses. Auto-__eq__ silently assumes its fields return a real bool from ==. Arrays don't. Three ways out, in order of preference:

  • Mark array fields field(compare=False) and assert equality explicitly in the test with np.array_equal(a.positions, b.positions) — clearest about intent.
  • Write a custom __eq__ (drop the generated one with @dataclass(eq=False)) that uses np.array_equal per array.
  • Compare the plain scalar fields via the dataclass and the arrays separately in the test.

This is the deep-module lesson in miniature: the decorator's convenience holds until a field's == doesn't return a bool — then you must take the wheel. Knowing where the generated code stops being right is the whole skill.

7 · Micro-exercise 5 minutes

No new deliverable — this is a warm-up you fold straight into your 1.1 World. In a scratch file or the REPL:

  • Write a @dataclass World with ids and positions NumPy-array fields plus an occupancy_cap: int, using field(default_factory=…) for the arrays.
  • Construct two identical worlds and run world_a == world_b. Watch it raise the "ambiguous truth value" ValueError — feel the trap from §5 first-hand.
  • Fix it: mark the array fields compare=False, and assert equality with np.array_equal. Confirm == now returns a real bool.
  • Add a __post_init__ that builds an _id_to_row dict (an init=False field). Verify it lands after construction.
  • Flip to @dataclass(frozen=True, slots=True). Confirm world.occupancy_cap = 9 now raises — then confirm world.positions[0] = 99 still succeeds (shallow freeze, §3).
Then carry it into 1.1. You don't need to report back on this one separately — bring it into your world.py. When you paste 1.1 for review I'll be checking exactly these: is World a dataclass (not Pydantic)? do the array fields dodge the __eq__ trap? is the id→row map derived in __post_init__ rather than passed in? Get these right and the determinism/invariant tests in 1.1 fall out cleanly.
